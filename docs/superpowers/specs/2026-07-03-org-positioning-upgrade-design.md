# 定位升级方案：一人公司组织架构 v1（薄公司 · 指挥官—参谋长制）

- 日期：2026-07-03
- 状态：待 Owner review
- 性质：真源专属方案文档（不入下游同步范围）
- 背景：Owner 提出把「一人公司」的组织架构框到整个生态之上——本人为 CEO，开发团队工作流为开发部门的 SOP，生态层设一个负责人协调立项、调研与工作流同步。经探讨收敛为本方案。

## 0. 探讨过程与关键决策（留痕）

1. Owner 原始设想：CEO → 开发部门 → 项目经理（生态层）+ 各开发组。
2. 澄清一：生态根**不是项目经理**，是研发部门负责人；组内不设 PM（半自动工作流，Owner 亲自进组行使项目级 PM 职权）；负责人不做各项目 PRD/架构/开发/部署，只做立项调研、建项目、协调工作流同步。
3. 澄清二（coordination 调研结论）：coordination 是跨项目公告板 + 档案室；需求提报/承接本来就是各项目会话**授权直写**，不依赖专职会话。真正需要专职会话的只剩 PROJECTS.md 同步等少量维护活。Owner 接受「仓库保留、专职会话撤销、维护职责并入部门负责人」。
4. 澄清三（组织隐喻探讨）：对照业界三种 AI 组织原型——虚拟公司制（MetaGPT/ChatDev）、群聊广播制（AutoGen，P12 已否）、图编排制（LangGraph/CrewAI，属未来全自动形态）。结论：对话型 Agent 穿公司外衣最稳，但只抄「薄」的——**方案 3 结构 + 方案 1 词汇温度**，Owner 拍板采纳。

## 1. 定位声明

- **升级前**：一套面向「一人公司」的 AI 编程助手多角色开发团队工作流（视角限于单项目内的团队）。
- **升级后**：**一人公司的 AI 组织操作架构**。以 Owner 为唯一指挥官（CEO），参谋长打理生态事务，各项目组按统一 SOP 作战，公告板承载跨组沟通；原「开发团队工作流」重新定位为**每个作战组的标准作业规程（SOP）**，是公司架构的一层而非全部。
- **对外品牌不变**：README 对外表述仍是「一人公司 AI 开发团队工作流」；「指挥官—参谋长」是内部架构定义。

## 2. 组织架构与职责

```text
指挥官（Owner / CEO）—— 唯一决策者
├── 参谋长（生态根会话 @ niuma-cheng/）
├── 公告板（coordination 仓 —— 场所，不是人）
├── 工作流组（agent-workflow —— 造 SOP）
└── 项目组 ×N（xiaobao / ai / workboard —— 用 SOP 作战）
```

| 座位 | 物理位置 | 职责 | 明确不干 |
|------|----------|------|----------|
| 指挥官 | Owner 本人 | 一切拍板；半自动阶段亲自进各组行使项目级 PM 职权 | —— |
| 参谋长 | 生态根会话（`niuma-cheng/` 根目录） | ① 立项受理+建壳 ② 维护公告板（PROJECTS 同步、台账勾选、状态落地）③ 协调工作流回流（跑 sync 脚本）④ 陪指挥官调研 + 生态导航索引 | 不做任何项目的 PRD/架构/开发/部署；不替指挥官决策；不进组扮演工作流角色 |
| 公告板 | coordination 仓 | 需求池 / BCR 池 / 状态台账 / 契约 / 决策的物理载体 | 撤销专职守门会话；各组授权直写协议**完全不变** |
| 工作流组 | agent-workflow 真源仓 | 演进 SOP；评估落地 BCR | 不管生态事务 |
| 项目组 | 各项目仓 | 按 SOP 作战，组内 PM/Architect/Developer/DevOps，无组长 | 不跨组直改别组的仓 |

### 三条组织法则（原则层）

1. **结构跟着物理走**：一个头衔 = 一个目录 + 一个会话；无座位不发头衔，任何没有座位的头衔都是管理戏服，禁止发放。
2. **参谋不决策**：参谋长只出方案、办事务、传信息；拍板永远在指挥官。
3. **半自动是设计选择而非缺陷**：指挥官亲自进组当 PM 是本阶段形态；全自动（决策权下放）留给下一次定位升级，本次不做。

## 3. 机制变更（本次升级实际改变行为的三处）

### 变更 ①：撤销 coordination 专职会话，维护职责并入参谋长

- 现行 baseline（`cross-project-collaboration.md` §生态参与者与跨界协议）里「协调真源台账」是四类参与者之一（一个会话节点）；升级后它降为**场所**，参与者从四类变三类 + 一个场所：框架真源 / 开发型下游 / 参谋长（原「生态索引根」升格，吸收协调台账维护岗）+ 公告板（场所）。
- 「谁能写什么」矩阵：参谋长对协调台账从「只读 + 回执」变为 **✅ 直写**（逐 commit 留痕）。
- 元信息同步从「三方接力」（子项目登记 → coordination 会话改 PROJECTS → 根会话订正索引）简化为**两方接力**：子项目登记 → 参谋长一站式（改 PROJECTS + 订正索引 + 勾两列）。
- 各项目直写提报/承接（REQ/BCR/状态/台账行）**不变**。

### 变更 ②：参谋长获得第二个跨界例外——工作流回流执行权

- 真源 baseline 合 main 后，由参谋长对各下游跑 `sync-downstream.sh` + 逐仓 commit/push + 回填 BCR 回流清单。
- 约束：只准经脚本同步框架文件，**禁止手改下游任何文件**；每轮回流开始前向指挥官报备一句、结束后汇报回执。
- 原第一例外「立项一次性建壳」不变。

### 变更 ③：生态根会话身份重写

- 从「只读导航/索引助手（+立项例外）」升格为「参谋长」。
- 根 `CLAUDE.md` 相应重写：身份定义、四项职责、两个例外、「参谋不决策」红线、公告板维护规程（含继承现有「真源可疑时停机」自检机制）。

## 5. 明确不做（本次范围外）

- 不做全自动化、不下放任何决策权。
- 不设组长、不新增任何部门/头衔。
- coordination 目录结构、需求流转协议（REQ 生命周期、授权直写、承接权限）一律不动。
- 各项目组内工作流（4 角色、门禁、Review）一律不动。
- workboard 不改代码（若其 UI 展示定位文案，仅数据配置层面跟进，不入本次范围）。

## 6. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 参谋长写权限扩大（coordination 直写 + 下游 sync） | 写公告板逐 commit 留痕可审计；下游只准经脚本、禁止手改；「参谋不决策」红线写进根 CLAUDE.md 首屏 |
| 「参谋长」角色语料不如「产品经理」丰富，行为唤起可能弱 | 根 CLAUDE.md 用职责清单钉死行为，不依赖头衔联想 |
| 撤销 coordination 会话后台账无人巡检 | 参谋长职责②内置「被唤起时自检台账在途行」，继承现有根索引自检机制 |

## 7. 与未来的接口

下一次定位升级（全自动化）的形态已预留：**指挥官逐步向各组下放战术决策权**（如组内恢复 PM Agent 并授予迭代启动权），组织词汇体系（授权、汇报、门禁）不需要更换；本方案的「薄公司」结构是其兼容基座。

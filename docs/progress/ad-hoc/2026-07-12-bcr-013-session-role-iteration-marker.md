## 背景（问题属实，已实证）

下游 workboard 要把「会话 ↔ 角色 ↔ 迭代」精准对应，目前只能靠启发式猜。核对消费方源码 `niuma-cheng-workboard/src/server/sync/session-meta.js`：

- **角色识别** `detectRole()`：① 首条用户消息命中 `你是XX` → 置信度 1.0；② 否则扫前 5 条用户消息做关键词加权 → 0.5~0.9；③ 打平不硬猜 → `Unknown`。
- **迭代识别**：**该文件完全没有迭代识别逻辑**。BCR 附 workboard Developer 预研：`vX.Y` 在会话开头仅 3%、标题 0 次、全文 71% 但无法定位到具体迭代。

**已实证的数据管道前提**（`claude-sync.js`）：`288-290` 行 user 与 assistant 消息**都已 parse 入库**（存进 `messages`）；但 `299-307` 行 `userContents` 只收 user 消息前 5 条，`detectRole` 只喂它——**assistant 消息存了却没参与识别**。含义：workboard 若要多用一个信号源，**改的只是 `detectRole` 的输入，数据管道不用动**。

结论：角色信号仅当用户恰好打了「你是XX」才可靠，迭代维度几乎无信号，**准确度天花板卡在会话源头**——但下游解析侧仍有一块**未挖的存量**（已入库的 assistant 消息）。

## 评估

**1. 归口正确。** target = agent-workflow，属框架层议题，落职责③（BCR-009 起参谋长兼评估落地）。

**2. 「不为单一下游耦合」这条张力——比初版更严格地看：** 一条机器可读的「本会话=什么角色·哪个迭代」标识，只有当它**落进框架自己的产物（迭代记录）**时，才真正服务框架内生需求（流程审计直读落盘产物，不读会话转录）。**读会话转录里 assistant 播报那一行的，目前只有 workboard。** 所以：

- **落盘形态（写进迭代记录 / 映射真源）= 框架原生**，审计直接受益，自证成立。
- **assistant 播报一行 = 主要仍是喂 workboard 的实时可见性**，自证偏弱——这点初版高估了，本版纠正。

**3. 设计岔路——marker 落点，四选项（按「越靠落盘越确定、越服务审计」排序）：**

| 方案 | 形态 | 确定性 | 服务对象 | 代价/风险 |
|------|------|--------|----------|-----------|
| **C 落盘「时间戳+角色+迭代」映射真源** | 启动/收尾机制在 `docs/progress/` 写映射行（提报方明细候选②的改良：不依赖 session id，用时间戳） | **字段确定**；绑定：有 session id 则确定，走(项目,时间窗,角色)匹配则**同角色并发会话有撞窗歧义** | 框架 + 审计原生 | 需验证：会话能否在启动时可靠写盘；并发撞窗如何消歧 |
| **D 收尾把角色+迭代写进迭代记录** | `mechanisms.md` 收尾机制写入（落盘产物） | **确定**（用户确认收尾时写） | 框架 + 审计原生 | **只覆盖走到收尾的会话**；中途放弃/进行中会话无记录；滞后 |
| **B assistant 启动输出固定格式标识行** | 「工作流启动」后助手输出一行（角色+迭代，迭代自 `INDEX.md` 派生） | **非确定**——LLM 每轮是否精确吐固定格式不保证（会漏/变体/被上文带偏）；**是「高置信度启发式」，强于现状，但不是 1.0**。要真确定需 hook/模板强制输出（另一份成本） | 主要喂 workboard 实时可见性 | 下游只需改 `detectRole` 输入源（管道不动）；但信号本身不硬 |
| A 用户手打结构化触发 | 要求用户开场打 `角色=… 迭代=…`（明细候选① = A 的改良：复用「你是XX」+补迭代号） | 确定（用户输入） | workboard | **违背自然语言入口**、迭代号仍要用户手打易漏——不建议 |

**关键纠正**：初版把 B 说成「确定性 1.0」是错的。现有代码给 1.0 是因为信号源是**用户确定输入**（「你是XX」）；B 把信号源换成 **LLM 自述**，模型输出本身不确定。真确定的信号只在「用户输入」或「落盘文件」，不在「模型自述」。

## 一个未点破的共性局限（四方案都受制）

「一会话 = 一角色一迭代」是**静态快照假设**，与真实会话动态不符：

- 一个会话可能先关 `v0.2` 再启 `v0.3`（跨迭代）；
- 参谋长 / General 会话不属任何迭代；
- 迭代可能在会话中途推进。

B、D 实际是「两端采样」（启动播报一次 / 收尾落一次），**中间段变更是盲区**。C 若在每次角色/迭代切换时补写映射行可缓解，但也更重。**任何方案都不是全覆盖**——Owner 决策时须知此非「一次标识解决一切」。

## 建议（经 review 修订，待 Owner 拍板）

**不再是「直接采纳 B」。按成本从低到高分档，建议先探底再决定是否动框架：**

**第 0 档（推荐先做，零框架改动）**：让 workboard 在**解析侧**先挖存量——`detectRole` 扫已入库的 assistant 消息、改进启发式，实测缺口到底多大。这正是「框架改动不为单一下游耦合」推到底的结论：**缺口未证实材质之前，不给框架加永久约定**。这一档不需要本 BCR 落地框架，归 workboard 迭代。

**若第 0 档证明缺口仍显著，再进框架改动，且排序应是：**

1. **D 为主力**（落盘、确定、审计原生）——框架本该做：收尾把角色+迭代写进迭代记录。
2. **C 待验证候选**——若「启动可靠写盘 + 并发消歧」验证通过，C 是唯一「字段确定 + 结构化 + 审计原生」形态，可升为主。
3. **B 为增量**（且如实标注为「高置信度启发式，非 1.0」）——只在需要**进行中会话实时可见性**时加，或上 hook/模板强制才谈「确定」。

**token 契约（无论 B/C/D 哪条落地都要先定）**——格式必须满足两条硬约束，写进契约要求：

- (a) **抗误命中**：标识独占一行、强定界（如 `<!-- wf-meta: role=… iter=… project=… -->`），不能被正常散文撞上；
- (b) **确定性可正则**：字段与格式固定、可稳定正则解析。

契约落点建议 coordination `contracts/`（跨项目契约正规位），框架侧不先于契约落地。

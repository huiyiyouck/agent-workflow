## 背景（问题属实，已实证）

下游 workboard 要把「会话 ↔ 角色 ↔ 迭代」精准对应，目前只能靠启发式猜。核对消费方源码 `niuma-cheng-workboard/src/server/sync/session-meta.js`：

- **角色识别** `detectRole()`：① 首条用户消息命中 `你是XX` → 置信度 1.0；② 否则扫前 5 条用户消息做关键词加权 → 0.5~0.9；③ 打平不硬猜 → `Unknown`。
- **迭代识别**：**该文件完全没有迭代识别逻辑**。BCR 附 workboard Developer 预研：`vX.Y` 在会话开头仅 3%、标题 0 次、全文 71% 但无法定位到具体迭代。

结论：角色信号仅当用户恰好打了「你是XX」才可靠，迭代维度几乎无信号，**准确度天花板卡在会话源头**，下游解析侧再优化也到顶。

## 评估

**1. 归口正确。** target = agent-workflow，属框架层议题，落职责③（BCR-009 起参谋长兼评估落地）。

**2. 核心张力——须先过这一关：** 提案主要动机是「喂 workboard 的解析器」。框架改动不应为单一下游工具耦合。**采纳的前提是它能自证框架价值**：一条机器可读的「本会话=什么角色·哪个迭代·哪个项目」标识，同样服务于**流程审计**（`mechanisms.md` 流程审计要追「哪个会话在哪个迭代做了什么」）和会话可追溯性——这条立得住，则采纳；立不住，则只是工具耦合，应退回 workboard 自己在解析侧治标。**本评估认为可自证**（审计/追溯是框架内生需求），故建议往采纳方向走，但落地形态要压到最小。

**3. 真正的设计岔路——marker 放哪里？** workboard 消费的是会话转录里的**用户消息**。标识落点决定可行性：

| 方案 | 形态 | 优点 | 代价/风险 |
|------|------|------|-----------|
| A 用户手打结构化触发 | 要求用户首条打 `角色=PM 迭代=v0.3` 之类 | 落在用户消息，workboard 现成解析 | **违背自然语言入口**、易漏易错、把机器负担压给人——不建议 |
| B assistant 启动播报一行确定性标识 | 「工作流启动」加载 runtime 后，助手输出固定格式一行（角色+迭代+项目，迭代自 `INDEX.md` 自动取） | 不加重用户；迭代可从 `INDEX.md` 自动派生；确定性、置信度 1.0；对审计同样有用 | workboard 需扩展 `session-meta.js` 去解析 **assistant 消息**（当前只读用户前 5 条）——但这是下游解析侧小改，非框架负担 |
| C 启动时落盘 marker 文件 | 会话启动写 `docs/progress/.session-marker` 之类 | 结构化最干净 | **会话 ↔ 文件的绑定是 workboard 解决不了的难点**（哪个 session id 对哪个 marker），治标不治本——不建议 |
| D 收尾/迭代记录里带标识 | 在 `mechanisms.md` 收尾时把角色+迭代写进迭代记录 | 与现有产物合流 | 只覆盖「走到收尾」的会话，进行中/未收尾会话仍无信号；滞后——可作 B 的补充，不作主方案 |

## 建议（待 Owner 拍板）

**建议采纳方向 = 方案 B 为主、D 为辅，且分两步、以跨项目契约先行：**

1. **框架侧（本仓落地）**：在「工作流启动」流程（入口 `CLAUDE.md`/`AGENTS.md` 第 4 步之后 + `runtime.md` 启动段）新增一条约定——加载 runtime 后，助手以**固定格式**输出一行会话标识：角色、当前迭代（自 `INDEX.md`「当前迭代」自动取，无则 `无`）、项目名。作为 baseline 约定，回流全下游。
2. **下游 workboard 侧（归 workboard 迭代）**：`session-meta.js` 扩展解析该标识（含 assistant 消息），命中即置信度 1.0 且补上迭代维度；未命中回退现有启发式。

**关键前置——标识 token 格式是跨项目契约，须框架 + workboard 共同定稿**（建议登记 coordination `contracts/` 或在 BCR-013 明细里定死字段与格式），否则框架落一个 workboard 解析不了的格式，两头对不上。契约未定，框架侧不先落。

**明确不采纳**：方案 A（用户手打，违背自然语言入口）、方案 C（落盘 marker，绑定难点无解）。

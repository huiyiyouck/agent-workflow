# 基线修正提案：communications 命名轴 —— 按项目对 vs 按需求

- 日期：2026-06-22
- 模式：Proposal（基线修正候选）
- 来源：`niuma-cheng-xiaobao` 对齐真源（P7 第 3 步）时，下游工作区一处未提交改动
- 归属处理：真源维护方 + Owner 确认（注：提出时 xiaobao 的 WM 角色正被裁撤，本提案直接落真源 progress，不走下游 WM 归集）
- 当前状态：⏳ 待评议
- 影响文件：`docs/baseline/cross-project-collaboration.md` §communications

## 1. 背景

`cross-project-collaboration.md` 当前真源版（`006bb24`，P5）对 `communications/` 采用**按项目对**模型：

> 每一对存在协作的项目共用一份沟通文档，按项目对命名 `communications/{project-a}__{project-b}.md`（字典序），承载这对项目之间**所有**跨项目需求从承接到关闭的沟通过程。

xiaobao 在对齐真源前，本地工作区有一处未提交改动，把同一节改成**按需求**模型：

> 每个被承接的需求对应一份沟通文档，按需求命名 `communications/{REQ-id}-{短名}.md`，一个需求一份。

该改动已随本次对齐被丢弃（真源版覆盖），此处仅留痕其设计要点，供真源评议是否采纳。值得注意：xiaobao 的改动里带的「创建时机 / 职责分工」措辞与真源新版一致，说明它是**在看过真源新思路后有意翻转命名轴**，是一个深思熟虑的提案，而非旧版残留。

## 2. 两个模型对比

| 维度 | 按项目对（真源现状） | 按需求（xiaobao 提案） |
|------|---------------------|----------------------|
| 命名 | `{a}__{b}.md` | `{REQ-id}-{短名}.md` |
| 粒度 | 一对项目共用一份，承载该对所有需求 | 一个需求一份 |
| 文件数 | 有界（最多 C(N,2) 对） | 随需求量线性增长 |
| 与 REQUESTS.md 关系 | 一份文档多个 REQ-id，靠条目标注区分 | 与需求 id 一一对应，反查直达 |
| 长期可读性 | 单份会越积越长、跨需求混杂 | 需求关闭即归档，边界清晰 |
| 「这对项目之间的事」 | 一个文件全在 | 分散在多份 |

## 3. 取舍判断

两者都站得住：真源的按项目对**文件数有界**、同对项目历史集中；xiaobao 的按需求**边界清晰、与需求池一一对应**，在需求量上来后单份按项目对文档会变得臃肿。这是真源维护方的设计决策，不在下游对齐阶段单方定夺。

倾向（供参考，非结论）：需求量小时按项目对更省心；需求量大、跨项目需求频繁时按需求更可维护。可考虑的折中是真源保留按项目对为默认，但在文档里点明「需求量大时可改按需求」，把轴的选择留给生态规模。

# BCR-011/012 合并落地方案：迭代门禁结构收敛

- 日期：2026-07-08
- 模式：Workflow Audit / BCR 评估落地
- 执行身份：参谋长（agent-workflow 第二工作目录）
- 关联：coordination `REQUESTS.md` BCR-011、BCR-012
- 状态：review 应修已处理，真源改动随本轮提交落地

> 本文是评估和落地游标；BCR 状态真源仍以 coordination `REQUESTS.md` 为准。

## 背景

workboard v0.2 时间轴解析暴露两个同源问题：

1. 新迭代记录的阶段状态词表虽在 baseline 中存在，但 `docs/templates/iteration.md` 模板本身未显式约束，容易继续写出 `已完成`、`部署通过`、`已Review` 等非标准阶段状态。
2. 旧 UI/测试阶段已由 BCR-004/006 废弃，但部分口径仍可能让新记录误把废弃阶段当标准阶段或门禁标题。

## 评估结论

- BCR-011：采纳。模板需要直接声明阶段门禁状态枚举、部署检查状态枚举、最新轮次置末行、门禁区只放标准阶段。
- BCR-012：部分采纳。`iteration.md` 当前已经没有“测试阶段 / UI 方案阶段”小节，不需要按原描述删除；但“固定标准阶段字段集”的方向成立，并入 BCR-011 同轮落地。
- 历史迭代记录不回改。历史不一致由读取侧兼容，真源只约束新迭代记录。

## 拟落地范围

1. `docs/templates/iteration.md`
   - 固定阶段门禁区只允许：PRD 阶段、设计阶段、实现阶段、部署就绪检查。
   - 明确阶段门禁状态：`待Review / Review中 / 修改中 / 已定稿 / 阻塞 / 已跳过`。
   - 明确部署检查状态：`待检查 / 检查中 / 通过 / 阻塞 / 已跳过`。
   - 明确每张门禁表最新轮次置末行。
   - 明确停放性记录、未来方向、讨论记录不得作为 `## 阶段门禁` 下的三级标题。
   - 概览「当前阶段」同步使用 `部署就绪检查 / 迭代关闭`，避免顶部枚举和门禁标题分叉。
   - PRD 初始 R1 状态从 `Review中` 调整为 `待Review`，符合“初版刚提交，尚无人 Review”的词表含义。
2. `docs/baseline/standard-iteration-quick.md`
   - 速查层同步模板约束，避免只在完整规范里才可见。
3. `docs/baseline/role-architect.md`
   - 清理 BCR-004 后残留的 UI 阶段检查口径，改为确认 PRD（含界面要点）已定稿。
4. `docs/baseline/role-devops.md`、`docs/baseline/multi-agent-workflow.md`
   - 收敛阶段命名：`部署阶段 / 部署` 改为 `部署就绪检查`。
   - 不改 `role-devops.md` 的「部署通过 / 部署阻塞 / 跳过部署」状态表；该表是 DevOps 叙述层状态，不作为迭代记录门禁枚举。

## Review 自检点

- `iteration.md` 不新增 UI/测试阶段，也不恢复旧阶段。
- 速查、模板、角色手册的标准阶段口径一致。
- `待Review / Review中 / 修改中 / 已定稿 / 阻塞 / 已跳过` 只用于阶段门禁；部署检查继续使用独立状态枚举。
- 不回改下游历史迭代记录。

## Review 结论（2026-07-08）

- Review 方：参谋长（受指挥官指派做落地前自检）。总体：**方案成立、可落地**，实证核对通过。

### 已核实通过

- BCR-012「部分采纳」依据属实：`iteration.md` 现存门禁小节仅 `PRD 阶段 / 设计阶段 / 实现阶段 / 部署就绪检查`，无「测试阶段 / UI 方案阶段」，原提案的“删除”在现行 main 无对象。
- 阶段门禁词表 `待Review / Review中 / 修改中 / 已定稿 / 阻塞 / 已跳过` 在模板 / `standard-iteration-quick.md` / `role-architect.md` 三处逐字一致。
- 部署检查独立枚举 `待检查 / 检查中 / 通过 / 阻塞 / 已跳过` 在模板与速查一致。
- `role-architect.md` UI→PRD 清理符合 BCR-004。
- 回流下发面（`templates/` + `baseline/`）无迭代门禁 UI/测试阶段残留：grep 命中项均为「已废弃」说明或合法的非迭代 UI Concept 工作模式。
- coordination `REQUESTS.md` 评估回填与本方案结论一致。

### 应修（1 处，已处理）

### 追溯提示（不强求）
